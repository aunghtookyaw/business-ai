Executive Conclusion

BigShot AI Business OS is operational but not yet production-stable. The current system supports finance reporting, payment receipt processing, NocoDB data management, local AI analysis, and Telegram delivery. Its principal risks arise from duplicated payment state, overlapping KPI paths, weak database enforcement, development-grade service hosting, and deployment of uncommitted code.

The largest architectural risks are:

  1. Payment truth is duplicated between Payment_Receive and transaction summary columns.
  2. The deployed runtime executes uncommitted working-tree code.
  3. KPI calculations have multiple overlapping entry paths and adjustment layers.
  4. Database relationships are not protected by foreign keys.
  5. The payment application runs Flask’s development server.
  6. Startup and restart controls depend on broad process-name termination.
  7. Existing backups predate the current code and database state.

Payment Architecture Finding

Payment data exists in two forms:

  • Event/history source: Payment_Receive.Receive_Amount
  • Materialized summary state: Total_Received, Outstanding_Balance and Payment_Status

This synchronization supports faster reporting but creates stale-value risk because history and summaries can be queried independently. A second legacy insertion path, payment_receive_insert(), performs similar calculations separately and requires Phase 2 investigation.

Log Evidence

Logs contain controlled rejection of overpayment attempts:

  • Voucher 77: calculated outstanding −6,800,000
  • Voucher 45: calculated outstanding −6,600,000

The requests returned controlled responses. Available evidence does not show these attempts crashing the payment service.

Hidden KPI Transformation

business_agent._combined_kpi() starts with kpi_overview, removes Sote Phwar income found in Transection, and adds Sote Phwar invoice income from Sotephwar_Transection. This prevents duplication but means executive KPI values can differ from direct kpi_overview() output.

Dual KPI Architecture

The legacy telegram_kpi_bot.py uses NocoDB-based KPI modules, while the preferred bot uses PostgreSQL formula_engine. If both bots run with the same token, they can conflict operationally and return differently calculated KPIs.
